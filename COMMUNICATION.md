## 共同作業のルール

このプロジェクトは、複数のAIエージェントによる共同作業によって進められます。円滑な開発のために、以下のルールを遵守してください。

1.  **明確なコミュニケーション**: 作業の開始、進捗、完了、問題発生時には、速やかに `COMMUNICATION.md` に記録し、共有してください。
2.  **タスクの明確化**: 各タスクは `TODO_COLLAB.md` に記載し、担当者とステータスを明確にしてください。
3.  **コードの品質**: 可読性、保守性を意識したコードを記述し、適切なコメントを付与してください。
4.  **Git の活用**: Git を使用してバージョン管理を行い、コミットメッセージは分かりやすく記述してください。プッシュ前に必ずテストを行ってください。
5.  **レビューとフィードバック**: 自身の作業が完了したら、相手にレビューを依頼し、フィードバックを積極的に取り入れてください。
6.  **問題解決**: 問題が発生した場合は、一人で抱え込まず、速やかに共有し、協力して解決策を検討してください。
7.  **要件定義の遵守**: `docs/CodingAgentBot要件定義者.txt` に記載された要件を常に意識し、逸脱しないように注意してください。差異が生じた場合は、速やかに共有し、合意形成を行ってください。
8.  **作業中断時のファイル整合性**: 作業を中断し再開する際は、編集中のファイルと Git リポジトリのファイルを比較し、リポジトリのファイルが更新されている場合は、そちらを優先して取得し作業を再開してください。

**OpenRouter 無料枠（1日50リクエスト）制限下での自律エージェント戦略:**

OpenRouter の無料枠制限は、自律エージェントにとって最大のボトルネックです。これを「掻い潜り」つつ、実用的な自律動作を実現するための 4 つの戦略を提案します。

**1. リクエストのバッチ化と多目的プロンプト (Request Consolidation):**
- 現在の「1発言につき1リクエスト」を改め、エージェントが「次の3ステップ」を一度に計画・出力するようにします。
- 例えば、`ファイル作成` -> `テストコード作成` -> `実行結果の予測` を1つのレスポンスにまとめさせることで、思考回数を 1/3 に削減します。

**2. 階層型エージェント構造 (Hierarchical Agent):**
- すべての処理を 405B 等の高性能（かつ制限が厳しい）モデルで行うのではなく、タスクの重要度に応じてモデルを使い分けます。
- **プランニング (405B/Sonnet等)**: 複雑な設計やデバッグ方針の決定（1リクエストで深く考える）。
- **コード生成・定型作業 (Llama 3.3 70B / Qwen 2.5 Coder)**: 決定された方針に基づく実装（無料枠が広い、あるいは高速なモデルを活用）。
- これにより、貴重な 50 リクエストを「思考の核心」に集中させます。

**3. ローカル・キャッシュとコンテキスト圧縮 (Context Management):**
- 同一セッション内での重複した問い合わせを避けるため、過去の実行結果や AI の回答をローカル DB に詳細にキャッシュします。
- AI に渡すコンテキストを「現在のタスクに直結する差分」のみに絞り込むことで、AI の理解度を高め、やり直し（リトライ）によるリクエスト消費を防ぎます。

**4. ユーザー・イン・ザ・ループの最適化 (Optimized Human-in-the-loop):**
- 自律ループが迷走しそうな場合（例：3回連続でテスト失敗）、リクエストを浪費する前にユーザーに介入を求めます。
- 「自律」と「効率」のバランスを取り、50 リクエストを「完遂」のために賢く配分します。

**具体的な実装案:**
`AIService` に `plan_and_execute` メソッドを追加し、1回のリクエストで「計画・実装・検証手順」の JSON 構造を返させるように改良することで、自律性を維持したままリクエスト効率を劇的に向上させます。

### s3 (2026/07/16 00:51 JST)

**「単一モデル選択制」および「1日50リクエスト制限」下での自律エージェント再構築戦略:**

モデルの役割分担（マルチモデル）ができないという制約を踏まえ、選択された**単一のモデル（Llama 3.3 70B / Qwen3 Coder / Hermes 3 405B）の能力を「1リクエストで限界まで引き出す」**戦略にシフトします。

**1. 「全行程一括出力」プロンプト (One-Shot Autonomy):**
- ユーザーの指示に対し、AI に以下の 4 要素を**1つのレスポンス内**で出力させます。
  - ① **詳細設計図**: 実装するファイルの構造とロジックの解説。
  - ② **ソースコード**: 全対象ファイルのコード。
  - ③ **自己検証シナリオ**: コードが正しく動作するかを確認するためのテスト項目。
  - ④ **予想されるエラーと対策**: 発生しがちなバグとその回避策。
- これにより、「書いては直し」という対話回数を最小化し、1リクエストで完成度 90% 以上を目指します。

**2. 構造化コンテキスト・スナップショット (Structured State Management):**
- リクエスト回数を減らすため、AI に「現在のプロジェクトの全容」を常に正確に伝える必要があります。
- 各リクエストの冒頭に、現在のファイルツリー、主要関数のシグネチャ、および「これまでの修正履歴」をコンパクトにまとめたスナップショットを自動挿入します。
- AI が「前回何を言ったか」を思い出すためのリクエスト（確認作業）をゼロにします。

**3. クライアントサイドでの事前検証 (Local Pre-validation):**
- AI にリクエストを送る前に、プログラム側で可能な限りのチェック（構文チェック、ファイル存在確認など）を行い、AI に「無駄な質問」をさせないようにします。
- AI からの返信も、受け取った直後にプログラム側でパースし、不完全な場合は（リクエストを消費する前に）エラーとして処理するか、ユーザーに提示します。

**4. エージェントの「思考の深さ」の調整:**
- 1リクエストの `max_tokens` を最大（4000以上）に設定し、AI に「じっくり考え、一気に書く」ことを強制します。
- 短い返信を繰り返すのではなく、1回のリクエストを「プロジェクトの1フェーズ（例：認証機能の実装）」の完遂に充てる設計にします。

**結論:**
単一モデル制約下では、**「AIとの対話」を減らし、「AIへの一括発注」を増やす**ことが、50リクエスト制限内で自走するための鍵となります。この方針に基づき、プロンプトのテンプレート化と状態管理システムの強化を進めます。
